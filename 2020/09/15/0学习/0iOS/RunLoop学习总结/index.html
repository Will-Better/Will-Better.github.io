RunLoop 学习总结

发表于 2020-09-15 更新于 2020-09-18 分类于 00-学习-iOS ， OC

1. RunLoop 简介

1.1 什么是RunLoop?

RunLoop是通过内部维护的事件循环来对事件或信息进行管理的一个对象。

  • 首先它是一个对象

  • 有消息或事件(eg：触摸事件、UI刷新事件、定时器事件、Selector事件)需要处理时，立刻被唤醒。 由内核态转换为用户态。

  • 没有消息需要处理时，休眠以避免CPU资源占用，提高程序性能。由用户态转换为内核态。


1.2 RunLoop与线程

一般来讲，一个线程一次只能执行一个任务，执行完成后线程就会退出。而RunLoop的机制就会使线程能随时处理事件但并不退出。

  • 从RunLoop的数据结构可得知，线程和RunLoop对象是一一对应的。
  • 只能在当前线程中操作当前线程的RunLoop，而不能去操作其他线程的RunLoop。
  • 主线程的RunLoop对象系统在UIApplicationMain函数内部自动创建。
  • 创建的线程并没有RunLoop对象，需要通过获取RunLoop来创建(系统不提供创建方法，当可以通过第一次获取线程的RunLoop方法让系统创建RunLoop)，它的销毁则是在线程结束的时候。

2. RunLoop的类和数据结构

我们主要了解一下Core Foundation框架下关于RunLoop的5个类。而Foundation框架下的NSRunLoop只是CFRunLoop的封装，提供了面向对象的API(但这些API不是线程安全的)。

  • CFRunLoopRef：代表RunLoop的对象,它提供的纯C函数的API是线程安全的。
  • CFRunLoopModeRef：RunLoop的运行模式
  • CFRunLoopSourceRef：就是RunLoop模型图中提到的输入源/事件源
  • CFRunLoopTimerRef：就是RunLoop模型图中提到的定时源
  • CFRunLoopObserverRef：观察者，能够监听RunLoop的状态改变

这5个类的关系图如下：

一个RunLoop对象（CFRunLoopRef）中包含若干个运行模式（CFRunLoopModeRef）。而每一个运行模式下又包含若干个输入源（CFRunLoopSourceRef）、定时源（CFRunLoopTimerRef）、观察者（CFRunLoopObserverRef）。

  • 每次RunLoop启动时，只能指定一种运行模式（CFRunLoopModeRef），这个运行模式（CFRunLoopModeRef）被称作CurrentMode。

  • 如果需要切换运行模式（CFRunLoopModeRef），只能退出当前Loop，再重新指定一个运行模式（CFRunLoopModeRef）进入，这样做主要是为了分隔开不同组的输入源（CFRunLoopSourceRef）、定时源（CFRunLoopTimerRef）、观察者（CFRunLoopObserverRef），让其互不影响 。

2.2 CFRunLoopModeRef

CFRunLoopModeRef 的主要数据结构大致如下：

启动App后， 系统默认会注册5种Mode:

  • kCFRunLoopDefaultMode：App的默认运行模式，通常主线程是在这个运行模式下运行的。(常用)

  • UITrackingRunLoopMode：界面跟踪Mode,用于 ScrollView 追踪触摸滑动，保证界面滑动时不受其他 Mode 影响。(常用)

  • UIInitializationRunLoopMode：在刚启动App时第进入的第一个 Mode，启动完成后就不再使用。

  • GSEventReceiveRunLoopMode：接受系统内部事件，通常用不到.

  • kCFRunLoopCommonModes：不是实际存在的一种Mode,相当于标记模式，是同步Source/Timer/Observer到多个Mode中的一种技术方案，很特殊。(常用)

涉及到的具体应用场景会在 4.RunLoop的具体应用场景 体现。

还有更多的苹果内部的Mode,但这些在开发中就很难遇到了。


2.3 CFRunLoopTimerRef

CFRunLoopTimerRef 是基于时间的触发器，它和 NSTimer是toll-free bridged(免费桥转换)的，可以混用，相当于就是NSTimer。其包含一个时间长度和一个回调（函数指针）。当其加入到 RunLoop 时，RunLoop会注册对应的时间点，当时间点到时，RunLoop会被唤醒以执行那个回调。


2.4 CFRunLoopSourceRef

CFRunLoopSourceRef是事件源，通过在实际应用中调用函数来分类可分为两类：

  • Source0 ：非基于Port，只包含了一个回调（函数指针），它并不能主动触发事件。需要手动将其标为待处理且 手动唤醒 RunLoop来处理它。

  • Source1：基于Port，包含了一个 mach_port 和一个回调（函数指针），被用于通过内核和其他线程相互发送消息。 主动唤醒 RunLoop的线程。


2.5 CFRunLoopObserverRef

CFRunLoopObserverRef 是观察者，每个 Observer 都包含了一个回调（函数指针），当 RunLoop 的状态发生变化时，观察者就能通过回调接受到这个变化。可以观察的状态改变的时间点有以下几种：

2.6 mode item 说明

上面的 Source CFRunLoopSourceRef、Timer CFRunLoopTimerRef、Observer CFRunLoopObserverRef 被统称为 mode item，一个 item 可以被同时加入多个 mode。但一个 item 被重复加入同一个 mode 时是不会有效果的。如果一个 mode 中一个 item 都没有，则 RunLoop 会直接退出，不进入循环。

3.1 RunLoop的模型

在每次运行开启RunLoop的时候，所在线程的RunLoop会自动处理之前未处理的事件，并且通知相关的观察者。

具体的顺序如下：

  1. 通知观察者RunLoop已经启动。

  2. 通知观察者即将要开始的定时器

  3. 通知观察者任何即将启动的非基于端口的源

  4. 启动任何准备好的非基于端口的源

  5. 如果基于端口的源准备好并处于等待状态，立即启动；并进入步骤9

  6. 通知观察者线程进入休眠状态

  7. 将线程置于休眠直到任一下面的事件发生：

    • 某一事件到达基于端口的源
    • 定时器启动
    • RunLoop设置的时间已经超时
    • RunLoop被显示唤醒

  8. 通知观察者线程将被唤醒

  9. 处理未处理的事件

    • 如果用户定义的定时器启动，处理定时器事件并重启RunLoop。进入步骤2
    • 如果输入源启动，传递相应的消息
    • 如果RunLoop被显示唤醒而且时间还没超时，重启RunLoop。进入步骤2

  10. 通知观察者RunLoop结束。


3.4 RunLoop 的底层实现

苹果官方将整个OSX/iOS 系统从上至下大致划分为4个层次：

  1. 应用层包括用户能接触到的图形应用，例如 Spotlight、Aqua、SpringBoard 等。

  2. 应用框架层即开发人员接触到的 Cocoa 等框架。

  3. 核心框架层包括各种核心框架、OpenGL 等内容。

  4. Darwin 即操作系统的核心，包括系统内核、驱动、Shell 等内容，这一层是开源的。

Darwin的核心架构如下:

在硬件层上面的三个组成部分：Mach、BSD、IOKit (还包括一些上面没标注的内容)，共同组成了 XNU 内核。

XNU 内核的内环被称作 Mach，其作为一个微内核，仅提供了诸如处理器调度、IPC (进程间通信)等非常少量的基础服务。和其他架构不同， Mach 的对象间不能直接调用，只能通过消息传递的方式实现对象间的通信。”消息”是 Mach 中最基础的概念，消息在两个端口 (port) 之间传递，这就是 Mach 的 IPC (进程间通信) 的核心。

为了实现消息的发送和接收，mach_msg() 函数实际上是调用了一个 Mach 陷阱 (trap)，即函数
mach_msg_trap()，陷阱这个概念在 Mach 中等同于系统调用。当你在用户态调用 mach_msg_trap() 时会触发陷阱机制，切换到内核态；内核态中内核实现的 mach_msg() 函数会完成实际的工作，如下图：

RunLoop 的核心就是一个 mach_msg() (见3.2 源码的第7步)，RunLoop 调用这个函数去接收消息，如果没有别人发送 port 消息过来，内核会将线程置于等待状态(主线程调用栈是停留在 mach_msg_trap())。

但当我们滚动 UIScrollView 时，会发现showInfo方法不打印了(NSTimer不工作了)，而我们离开 UIScrollView 时，showInfo方法就打印了。

这是RunLoopMode导致的:

  • 当我们不做任何操作的时候，RunLoop处于NSDefaultRunLoopMode下。NSTimer在此种模式下，所以正常工作。
  • 当我们拖动 UIScrollView 的时候，RunLoop就结束NSDefaultRunLoopMode，切换到了UITrackingRunLoopMode模式下， NSTimer 不在此种模式下，所以就不工作了。
  • 当我们 UIScrollView 的时候，RunLoop就结束UITrackingRunLoopMode模式，又切换回NSDefaultRunLoopMode模式，所以NSTimer就又开始正常工作了。

我们可以使用 kCFRunLoopCommonModes 来使 NSTimer 在这两种模式下都能正常工作。

kCFRunLoopCommonModes 其实不是一种真实的模式，而是一种标记模式，意思就是可以在打上Common Modes标记的模式下运行。
